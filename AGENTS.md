# AGENTS.md

Always-on rules for AI agents working in this repository.

## Surprise Handling

If you encounter ambiguous requirements, surprising behavior, or a situation where the "right" action is unclear, **stop and ask the developer** before proceeding.

## Project Overview

Spam blocker for the Bitsocial protocol — risk scoring, challenges, and rate limiting for pseudonymous communities.

## Instruction Priority

1. **User request** — explicit instructions in the current conversation
2. **MUST rules** — non-negotiable constraints below
3. **SHOULD rules** — strong defaults, override only with good reason
4. **Playbooks** — load-on-demand references for domain-specific context

## Stack

- **Runtime:** Node.js v22+ / TypeScript / ESM (use `nvm` to switch versions)
- **Server:** Fastify / better-sqlite3 / Zod
- **Workspace:** npm workspaces (3 packages: `server`, `challenge`, `shared`)
- **Tooling:** vitest / esbuild / Prettier / commitlint + husky

## MUST Rules

### Environment & Build

- Node.js v22+ required for all packages.
- Run `npm run build` after coding and make sure it passes. Run it for all packages.
- Both challenges and HTTP server/engine run in Node.js only, never in browser.
- Prefer static imports; dynamic imports should not be needed.

### Code Style

- Function parameters as a single object: `{param1, param2}` — more readable and easier to update.
- Everything should be properly typed. If you can't type something, ask the user for help.
- In user-facing UI text (iframes, HTML pages), use "Bitsocial" instead of "plebbit" and "community" instead of "subplebbit".

### Testing

- Add vitest tests for every new feature.
- Reproduce bugs with a test case first, then fix the code, then verify the test passes.

### Database

- No migration strategy needed — assume an empty DB.
- Prefer existing JSON columns over adding new tables/columns. If you still think a new table or column is needed, ask the user.
- If you add or modify a column in `packages/server/src/db/schema.ts`, remind the user to delete the existing database file. The schema uses `CREATE TABLE IF NOT EXISTS` which does not alter existing tables, so the old DB will be missing the new columns.

### Dependencies & Types

- DO NOT duplicate plebbit-js schemas or types — import them from plebbit-js.
- `subplebbit.address` can be a domain. To get its public key, resolve it using the plebbit-js library.

### Shared Code

- Schemas shared between the challenge package and HTTP server/engine go in the `shared/` directory.

### Security & Trust

- Author fields are user-provided and not to be trusted, **except** `author.subplebbit` which is generated by the subplebbit and can be trusted.
- Authors can spam with different signers — the protocol is pseudonymous.

## SHOULD Rules

- Keep `README.md` in sync with the implementation when you do a git commit.
- Update `packages/server/src/risk-score/RISK_SCORING.md` when risk score calculation changes.
- Run `npm run generate-scenarios` in `packages/server` when risk factors change to regenerate `RISK_SCORE_SCENARIOS.md`.
- When modifying risk factors, check whether the `generate-scenarios` script itself needs updating to pick up your changes.
